Mise en place du domaine Active Directory

Phase 1 — Les fondations de l'infrastructure BleuesLatitudes

🚧
Phase en cours. Le domaine bleueslatitudes.local est créé. Petit accroc à régler avant la suite : accès admin du domaine bloqué (je passe par le mode DSRM pour réinitialiser, détails plus bas). Une fois débloqué, je m'attaque aux OU, groupes et utilisateurs.

Contexte

Côté infra, BleuesLatitudes part de zéro : 25 postes en groupe de travail, aucun compte centralisé, aucune sauvegarde, aucune supervision. La toute première brique à poser, c'est un domaine Active Directory. Sans ça, rien d'autre ne tient debout : ni les permissions sur les fichiers (Phase 2), ni les GPO, ni le partage centralisé.

Objectifs

  • Installer et configurer Windows Server 2025 (SRV-DC01)
  • Configurer l'adressage IP fixe sur le réseau 192.168.10.0/24
  • Déployer le rôle Active Directory Domain Services
  • Promouvoir le serveur en contrôleur de domaine — forêt bleueslatitudes.local
  • Créer la structure organisationnelle (OU par service)
  • Créer les comptes utilisateurs et groupes
  • Joindre un poste client (PC-01) au domaine

Environnement technique

  • Hyperviseur : Hyper-V (Windows 11 Pro, hôte 24 Go DDR4)
  • Switch virtuel : BLEUESLAT-NET (Private)
  • VM SRV-DC01 : Génération 2 — 2 vCPU — 2 Go RAM — disque 60 Go — Windows Server 2025 Standard (Expérience utilisateur)
  • Adressage : 192.168.10.1/24, passerelle non définie (lab isolé), DNS 127.0.0.1

Avancement détaillé

✅ Étape 1 — Création de la VM SRV-DC01

VM créée sous Hyper-V (Génération 2) à partir de l'ISO Windows Server 2025. J'ai pris la 2025 plutôt que la 2022 pour une raison simple : autant me former sur la dernière version, c'est celle que je vais retrouver en entreprise.

✅ Étape 2 — Réseau et identité

IP fixe en 192.168.10.1/24, DNS pointé sur 127.0.0.1 (le serveur sera lui-même DNS une fois promu). J'ai renommé la machine en SRV-DC01 avant la promotion AD : un truc que j'ai retenu, c'est que renommer un contrôleur de domaine après coup, c'est galère. Mieux vaut le faire dans le bon ordre.

✅ Étape 3 — Installation du rôle AD DS

Installation du rôle AD DS depuis le Gestionnaire de serveur, puis promotion du serveur en contrôleur de domaine. Nouvelle forêt avec bleueslatitudes.local comme domaine racine, niveau fonctionnel Windows Server 2025. J'ai aussi défini le mot de passe DSRM (Directory Services Restore Mode), une sorte de filet de sécurité pour reprendre la main si un jour le compte admin du domaine pose problème. Spoiler : ça arrive plus tôt que prévu, voir juste en dessous.

⚠️ Point bloquant — Accès admin du domaine

Et là, première vraie difficulté : impossible de me connecter en tant qu'administrateur du domaine. Le mot de passe est refusé. Soit j'ai dépassé le seuil de tentatives et le compte est verrouillé, soit ce que j'ai noté ne correspond pas à ce que j'ai réellement saisi à la promotion (l'erreur est humaine).

Plan pour débloquer : redémarrage en mode DSRM, réinitialisation du compte Administrator via net user Administrator [nouveau-mdp], puis retour en mode normal. C'est exactement le scénario pour lequel le mot de passe DSRM existe — content de l'avoir défini, sans lui je serais bon pour repartir à zéro.

⏳ Étape 4 — OU, groupes et utilisateurs (à venir)

Une fois l'accès admin récupéré, je structure l'annuaire avec une OU par service (Direction, Bureau d'études / Commercial, Administratif), je crée les groupes de sécurité correspondants et les comptes des 25 employés. Cette étape est cruciale parce que tout ce qui vient après s'appuie dessus : les permissions sur le serveur de fichiers en Phase 2, les GPO ciblées, etc.

⏳ Étape 5 — Joindre PC-01 au domaine (à venir)

Création de la VM PC-01, config réseau, jonction au domaine. Le but, c'est de vérifier qu'un poste client peut s'authentifier sur le DC avec un compte AD — preuve concrète que tout fonctionne bout en bout.

Ce que je retiens

  • AD et DNS sont indissociables : un contrôleur de domaine est forcément serveur DNS de sa zone, ce n'est pas optionnel.
  • Promouvoir un serveur en DC, ça ne se défait pas à la légère. D'où l'utilité du mot de passe DSRM, justement (et là je suis bien content de l'avoir noté correctement, lui).
  • L'ordre des opérations a son importance : IP fixe → renommage → installation du rôle → promotion. Si on inverse, ce sont des manipulations correctives à faire derrière, jamais propre.
  • Le niveau fonctionnel (forêt, domaine) détermine les fonctionnalités dispo et les versions de DC tolérées dans la forêt. À garder en tête si on doit faire cohabiter plusieurs versions de Windows Server plus tard.

Compétences mobilisées

  • Installer et configurer un serveur Windows
  • Administrer Active Directory
  • Gérer les utilisateurs et les groupes
  • Configurer l'adressage réseau
  • Mettre en place une convention de nommage et un plan d'adressage
